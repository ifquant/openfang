## 第十章：未来之章——让 OpenFang 参与自己的进化

> 这一章不再把“自我演化”写成抽象方法论，也不把它写成战略报告。
> 它真正要回答的问题是：**基于 OpenFang 今天已经存在的代码底座，系统距离“受控参与自己的工程演化”还有多远，中间还差什么。**

如果前 0-9 章主要回答“OpenFang 现在是什么”，那么这一章要回答的是：

1. OpenFang 今天已经拥有哪些可以支撑演化闭环的真实能力。
2. 为什么“会改代码”不等于“可演化”。
3. 什么样的演化路径是可信、可审计、可回滚的。
4. 哪些事情现在可以做，哪些事情今天还不该做。

### 本章导航

这章按两条阅读路径组织：

1. **教程路径（本章）**：理解为什么可演化是一个工程问题，而不是模型炫技；理解 OpenFang 现有底座与现实边界。
2. **工程路径（附录 D）**：如果你准备把这里的判断落成 backlog、模板或规格，再去看 `openfang_tutorial_appendix_d_evolution_engineering.md`。

第一次阅读，建议按下面的顺序走：

1. 先看 10.0，明确这章真正讨论的问题。
2. 再看 10.1，理解 OpenFang 今天已经拥有的 4 个真实演化底座。
3. 再看 10.2，跟着一个完整案例走一遍受控演化闭环。
4. 最后看 10.3-10.5，理解现实边界、双轴校准和真正可执行的路线图。

### 10.0 为什么这章必须存在

到第九章为止，我们已经看到 OpenFang 有内核、有运行时、有工具系统、有工作流、有审计、有预算、有 API、有 CLI/TUI。问题在于：

**这些能力只是把 OpenFang 做成了一个强系统，还是已经把它推到“可以受控参与自己工程闭环”的门口？**

如果只是前者，那 OpenFang 仍然是一个静态系统。它会做任务，但不会持续改善自己。

可一旦要讨论后者，问题就立刻变硬了。这里真正要回答的，不是“模型能不能改几行代码”，而是下面 4 个更苛刻的问题：

1. 它怎么发现值得改的东西，而不是随机乱改。
2. 它怎么证明这次变化带来了收益，而不是只是把报错压下去。
3. 它怎么把每一步变化纳入审计链、预算门和回滚面。
4. 它怎么避免把“自我演化”做成无限自激的灾难回路。

这也是为什么第十章不能只写成“未来展望”。如果前九章讨论的是系统当前的器官，这一章讨论的就是：

**这些器官能不能被组织成长期代谢。**

#### 10.0.1 这章不是在问“AI 能不能写代码”

今天任何会产出 patch 的模型，都已经能在局部任务里写一点代码。但工程系统真正关心的不是“会不会写”，而是：

1. 能不能在真实信号驱动下提出变化。
2. 能不能在边界内执行变化。
3. 能不能在验证失败时及时停下。
4. 能不能让每次尝试留下下一轮可复用的证据。

从这个角度看，“自我演化”更接近一个受控工程闭环，而不是一次性的生成演示。

#### 10.0.2 先把这个话题放回工程史里

如果不把第十章放回更大的工程背景里，它很容易显得像一套作者自造术语。真正更稳的写法，是先承认这不是一个全新的问题，而是几个老问题在 LLM 时代重新汇合。

第一类先例，是 **self-hosting / bootstrapping**。

编译器编译自己、解释器运行自己、语言工具链逐步接管自己的构建过程，这些都说明“系统参与自己的生产过程”不是新鲜事。难点从来不是“能不能做”，而是“怎样把递归能力放进稳定的工程秩序里”。

第二类先例，是 **受控自动化维护**。

Dependabot、Renovate 这一类系统并不追求“自动重写一切”，而是专门做小步、低风险、可 review 的变更。这件事对第十章很重要，因为它提醒我们：真正能长期运行的自动化，不是权限最大，而是**爆炸半径最小**。

第三类先例，是 **大规模代码变更基础设施**。

Google 的 Large-Scale Changes 之所以重要，不是因为它“能改很多仓库”，而是因为它说明：只要想把变更做大，系统就必须先拥有更强的验证、审计、回滚和变更编排能力。没有这一层，所谓“大规模自演化”只会是危险词。

第四类先例，是 **验证技术而不是生成技术**。

Mutation testing、property-based testing、golden test、staged rollout 这些技术的共同点是：它们都在回答“怎么知道这次变化没有把系统悄悄改坏”。第十章真正欠缺的，恰恰不是更多“能写代码”的话术，而是更多“能判定变化值不值得留下”的知识。

最后还要明确一件事：**公开基准上的 LLM agent 表现，离“稳定接管跨文件工程维护”仍有明显距离。**

这不意味着路线错误，而是意味着第十章必须从一开始就承认现实边界：

- 大模型可以成为候选补丁生成器。
- 但它还不是可以直接接管主分支的自治工程师。

这也是为什么本章接下来要先回到 OpenFang 今天已经存在的真实代码，而不是继续堆抽象判断。

### 10.1 OpenFang 今天已经有什么演化底座

如果说前面的章节把 OpenFang 拆成了许多模块，那么第十章要做的第一件事，就是把这些模块重新组合成“演化底座”。

我认为，当前最值得抓住的真实底座有 4 个：

1. `WorkflowEngine`：负责把一轮变化组织成受控步骤。
2. `AuditLog`：负责把一轮变化写进可验证的审计链。
3. `MeteringEngine + ResourceQuota`：负责给变化过程加预算门。
4. `ToolPolicy`：负责给执行能力加权限门。

这 4 个组件今天已经存在于源码中。它们还没有自动组合成“演化系统”，但已经足以构成第十章最重要的现实起点。

#### 10.1.1 WorkflowEngine：OpenFang 已经会组织多步闭环，而不只是单步调用

这一点最值得先看 `openfang-kernel/src/workflow.rs`。

这个文件里最关键的几个落点是：

1. `WorkflowStep`，定义在 `workflow.rs` 的 L83 左右。
2. `StepMode`，定义在 L120 左右。
3. `ErrorMode`，定义在 L137 左右。
4. `WorkflowEngine`，定义在 L201 左右。
5. `execute_run()`，从 L414 左右开始。

如果你只记一件事，那就是：**OpenFang 当前并不是只有“让某个 agent 执行一次 prompt”这种原子动作，它已经有一个能把多步执行串成 run 的工作流壳。**

先看 `WorkflowStep`。它已经包含了：

1. `name`：步骤名。
2. `agent`：这一步交给哪个 agent。
3. `prompt_template`：允许把上一步输出或变量插进下一步。
4. `mode`：步骤执行模式。
5. `timeout_secs`：每一步的超时边界。
6. `error_mode`：失败后是终止、跳过还是重试。
7. `output_var`：把结果存进变量池，供后续步骤复用。

这里最值得教学的，不是“字段很多”，而是它已经把一轮复杂动作拆成了真正可编排的单位。

再看 `StepMode`。它已经支持：

1. `Sequential`
2. `FanOut`
3. `Collect`
4. `Conditional`
5. `Loop`

也就是说，今天的 WorkflowEngine 已经不是只有线性流水线，而是具备了：

- 条件分支
- 并行 fan-out
- 结果归并
- 有上限的循环

这对第十章的意义非常大，因为它直接决定了一件事：

**OpenFang 已经拥有了组织“观察 -> 生成 -> 验证 -> 总结”这类闭环的编排骨架。**

再看 `execute_run()`。它做的几件事特别像“演化任务的主心骨”：

1. 把 run 状态从 `Pending` 推到 `Running`。
2. 在多步执行中传递 `current_input` 和变量表。
3. 按 `ErrorMode` 决定失败、跳过还是重试。
4. 把每一步结果写进 `step_results`。
5. 在完成或失败后记录最终状态。

还有一个细节特别值得注意：`create_run()` 会保留运行历史，并在超出上限后淘汰最老的已完成 / 已失败 run。源码里这条上限是 `MAX_RETAINED_RUNS = 200`。

这说明 WorkflowEngine 的定位不是一次性 demo，而是默认已经考虑到“长期重复执行”的现实。

但它的边界也要讲清楚：当前 `WorkflowStep` 只有 `ById / ByName` 选 agent、`Sequential / FanOut / Collect / Conditional / Loop` 选执行模式，并没有“调用另一个 workflow”这样的步骤类型。因此它已经是一个可用的多步编排器，但还不是支持子工作流嵌套、层层复合的 workflow-of-workflows 系统。

如果把这一节压成一句话，就是：

**OpenFang 今天已经拥有“多步、带错误模式、带状态保留”的工作流编排器，它缺的不是闭环壳，而是更像演化任务的输入信号、验证门和治理语义。**

#### 10.1.2 AuditLog：它已经拥有一条可验证的审计链，而不是只会打印日志

第二个必须回到源码看的文件，是 `openfang-runtime/src/audit.rs`。

这里最关键的落点是：

1. `AuditEntry` 在 L28 左右。
2. `AuditLog` 在 L80 左右。
3. `record()` 在 L100 左右。
4. `verify_integrity()` 在 L141 左右。

这段代码最有教学价值的地方在于：OpenFang 的审计不是“写一行日志”那么轻，而是**把事件串成一条 SHA-256 hash chain**。

`AuditEntry` 当前至少包含这些字段：

1. `seq`
2. `timestamp`
3. `agent_id`
4. `action`
5. `detail`
6. `outcome`
7. `prev_hash`
8. `hash`

也就是说，每一条事件都不是孤立的。它同时携带：

- 自己的内容
- 前一条事件的 hash
- 由两者共同算出的新 hash

这条链的起点是 64 个 `0` 组成的 genesis sentinel。后面的每一条记录，都把前一条的 hash 带进来。

然后 `verify_integrity()` 会重新走完整条链，逐条校验：

1. `prev_hash` 是否连续。
2. 当前条目的 hash 是否能被重算出来。

如果任一条件不满足，它会直接返回 `Err(String)` 报告第一处断链位置，而不是静默继续。因此这里更准确的定义是：它已经拥有显式验链失败信号的审计底座，但还没有后续处置自动化。

这对第十章的意义非常直接：

**如果未来真的要让 OpenFang 参与自己的工程变更，那么每一次“为什么改、改了什么、结果如何”都应该能被写进这种可验证链，而不是散落在普通日志里。**

这里还有一个特别适合教学的收获：

第十章过去一直在谈“治理平面”“责任边界”，但没有把它落到真实代码。现在我们终于可以把那句话说实：

**OpenFang 今天已经有一条可以承载候选变更审计的底层结构。**

当然，它也有明显边界：当前 `AuditAction` 里已经有 `ToolInvoke`、`FileAccess`、`ConfigChange` 这类动作，但还没有“evolution attempt”这样的专用语义。因此第十章真正合理的工程判断，不是“审计还没有”，而是“审计底座已经有了，但演化任务的事件模型还没长出来”。

#### 10.1.3 MeteringEngine 与 ResourceQuota：它已经会给智能行为加预算门

第三个必须看的组合，是：

1. `openfang-kernel/src/metering.rs`
2. `openfang-types/src/agent.rs`

前者给出执行中的预算判断，后者给出 manifest 级别的资源配额形状。

在 `metering.rs` 里，最关键的落点是：

1. `MeteringEngine` 在 L18 左右。
2. `check_quota()` 在 L24-L57 左右。
3. `record()` 在 L19-L23 左右。
4. `check_global_budget()` 在 L59-L88 左右。
5. `estimate_cost_with_catalog()` 在 L196 左右。

而在 `agent.rs` 里，`ResourceQuota` 定义在 L243 左右。它当前包含的关键限制包括：

1. `max_memory_bytes`
2. `max_cpu_time_ms`
3. `max_tool_calls_per_minute`
4. `max_llm_tokens_per_hour`
5. `max_network_bytes_per_hour`
6. `max_cost_per_hour_usd`
7. `max_cost_per_day_usd`
8. `max_cost_per_month_usd`

第十章之前经常把“预算边界”写得很抽象，但这两段代码告诉我们一件更硬的事实：

**OpenFang 今天已经不是“理论上可以配预算”，而是已经有一个能按小时 / 天 / 月检查成本上限、还能结合模型价格估算成本的计量引擎。**

`check_quota()` 做的事情非常直接：

1. 查某个 agent 的小时花费。
2. 查日花费。
3. 查月花费。
4. 任何一项超线，都直接报 `QuotaExceeded`。

这就是第十章最该补回来的“硬知识感”。因为一旦要谈自我演化，最先要问的不是“模型够不够聪明”，而是：

1. 这轮尝试最多能花多少钱。
2. 最多能烧多少 token。
3. 最多能发起多少工具调用。
4. 超线时是降级、终止还是转人工。

而这些问题在 OpenFang 里，今天已经不是空话。

更重要的是，这里天然补上了第十章以前缺失的一条现实感：

**未来真正能长期运行的演化任务，不会是“放开模型狠狠干”，而会是“每一轮都带着预算门执行”。**

#### 10.1.4 ToolPolicy：它已经会做权限门，但还没到目录级写入治理

第四个必须看的文件，是 `openfang-runtime/src/tool_policy.rs`。

这里最关键的落点是：

1. `ToolPolicy` 在 L18 左右。
2. `resolve_tool_access()` 在 L76 左右。

这段代码的教学价值，首先在于它已经明确采用了 **deny-wins** 语义：

1. 先看 agent 级 deny。
2. 再看 global deny。
3. 如果配置了 allow 规则，就必须命中 allow 才能放行。
4. 子 agent 还要额外受深度限制。

`ToolPolicy` 当前支持的几类边界包括：

1. agent 级规则
2. global 级规则
3. tool group
4. subagent 最大深度
5. subagent 最大并发

这意味着第十章现在终于可以把“治理边界”落到一段真实行为上：

**OpenFang 已经拥有工具级、组级、层级深度级的访问控制。**

但这里同样要诚实：当前这层策略主要作用在**工具名**和**subagent 深度**上，还不是目录级文件写入 ACL。

这件事非常重要，因为它直接决定了我们在 10.2 要怎么设计案例。

过去第十章经常会写“限制 agent 只能改 docs 目录”。作为未来方向这没有问题，但就当前源码来说，这种目录级边界并不是 `ToolPolicy` 今天已经提供的能力。它今天提供的是：

- 哪类工具能不能用。
- 允许用到多深的子 agent。
- 是否采用 allowlist / denylist。

所以更诚实的工程判断应当是：

1. **工具级权限门今天已经有了。**
2. **目录级写入边界仍然需要和第三章的文件系统 / sandbox 治理一起补齐。**

这比空喊“全都能管”更重要，因为它直接决定第十章的可信度。

#### 10.1.5 把 4 个底座收成一句话

到这里，第十章终于可以不再悬空地说一句更硬的话：

**OpenFang 今天已经具备了组织演化闭环的 4 个基础构件：工作流编排、可验证审计链、预算门、工具级权限门。**

但它还没有自动长成“自我演化系统”。中间还差至少 4 层东西：

1. 稳定的问题发现信号源。
2. 演化专用的审计语义。
3. 更硬的验证模板。
4. 比工具级更细的写入边界。

这就把第十章从“未来畅想”拉回了更真实的位置：

**不是从 0 到 1，而是从 0.6 到 1。**

### 10.2 一个完整案例：教程行号漂移该怎样被受控修复

接下来不再讲抽象框架，而是走一个完整案例。

之所以选“教程行号漂移”而不是“自动修核心代码”，不是因为它更简单，而是因为它更符合第十章当前真正应该主张的 Generation 0：

**先从低风险、可验证、可回滚、可审计的维护劳动开始。**

假设场景是这样的：

1. 第三章引用了 `tool_runner.rs` 的某一段行号。
2. 上游代码重构后，行号已经漂移。
3. 教程里的描述还对，但落点已经失效。

这正是长期维护里最常见、也最适合自动化闭环接手的问题之一。

#### 10.2.1 第一步：先把“感觉不对”变成稳定信号

这一轮演化的起点，不应该是“模型觉得这里可能有问题”，而应该是一个稳定信号源。

例如：

1. 一个教程校验脚本发现引用行号不存在。
2. 一个文档检查任务发现文件路径还在，但段落已移位。
3. 一次 CI 检查发现教程中的函数名已不匹配当前源码。

今天的 OpenFang 仓库里还没有一个现成的“教程漂移探测器”，这恰恰是第十章真正该诚实指出的第一层缺口。

也就是说，当前最合理的 Generation 0 起点不是“让系统自己随机读教程找错”，而是：

**先补一个把漂移问题变成结构化信号的小工具或检查器。**

#### 10.2.2 第二步：判型，不要一上来就写 patch

一旦信号出现，系统先要判型：

1. 这是代码 bug，还是文档漂移。
2. 这是低风险单文件修正，还是会影响多章叙事的一致性变更。
3. 这是能自动生成候选补丁的问题，还是需要先请人工确认定位。

在这个案例里，最合理的判型结论通常是：

- 低风险
- 文档类
- 局部可验证
- 适合进入 Generation 0 自动修复闭环

注意这里最重要的不是“判断得多聪明”，而是：

**系统开始学会不给所有问题同样权限。**

#### 10.2.3 第三步：用 WorkflowEngine 把一轮尝试收成 run

到了这里，WorkflowEngine 才真正登场。

这一轮 run 完全可以被压成 4 步：

1. `detect`：读取教程和源码，确认引用失效的具体位置。
2. `propose`：生成一个最小补丁，只改失效引用，不改周边叙述。
3. `verify`：重新跑文档校验或至少做一次引用一致性检查。
4. `summarize`：输出这次修复的结果、证据和残余风险。

如果把这 4 步映射回 `workflow.rs`，你会发现它不是空想：

1. 每一步都可以是一个 `WorkflowStep`。
2. `detect` 的输出可以写进 `output_var`，供 `propose` 使用。
3. `verify` 可以设置更保守的 `ErrorMode::Fail`。
4. `summarize` 可以在最后聚合结果。

也就是说，第十章现在终于可以把“演化闭环”从名词压成一个真实可执行的骨架：

**它首先是一个 workflow run，而不是一个自由发挥的智能代理。**

#### 10.2.4 第四步：这轮 run 必须自带审计链

只要涉及候选变更，AuditLog 就不应该是可选件。

在这个案例里，最少也该记录这些事件：

1. 谁发起了这次检测。
2. 发现了哪一处引用漂移。
3. 生成了什么候选补丁。
4. 验证是通过、失败还是证据不足。
5. 最终是自动落地、转人工 review，还是放弃。

当前 `AuditAction` 还没有“evolution attempt”这种专用分类，这说明第十章真正该给出的行动项不是“审计已经完备”，而是：

**为演化类任务补一套专门的审计语义。**

这样做的好处，不是为了显得正规，而是因为自我演化一旦出问题，最先要问的永远是：

1. 问题是哪一轮引入的。
2. 当时证据是什么。
3. 为什么系统会认为这次变化值得接受。

没有这条链，任何“自我改进”都无法长期被信任。

#### 10.2.5 第五步：给这轮尝试明确预算和停止条件

这一轮案例绝不应该是“让模型自己试到成功为止”。

更合理的做法是，在 agent manifest 或演化任务配置里给它明确的小预算，例如：

1. 限定小时级成本。
2. 限定 token 上限。
3. 限定工具调用频率。
4. 限定失败重试次数。

这件事之所以重要，是因为文档修复看起来无害，但一旦信号源脏、补丁生成不稳、验证条件模糊，系统就可能进入无意义重试。

`MeteringEngine` 与 `ResourceQuota` 让我们今天已经可以把这层边界写成真实配置，而不是只在教程里口头要求“节制”。

#### 10.2.6 第六步：今天真正的权限边界该怎么设

这里必须非常诚实。

如果只看 `ToolPolicy`，OpenFang 今天已经能做到的是：

1. 允许文档修复 agent 使用某些低风险工具。
2. 禁止它调用高风险工具。
3. 限制它生成过深的子 agent。

但它今天还不能仅靠 `ToolPolicy` 就做到“只允许改 docs 某个子目录”。

所以在当前版本里，更可信的安全口径应该是：

1. 给文档修复任务单独 agent 配置更小的工具面。
2. 把自动修复先限制在低风险工作区或专门分支上。
3. 把最终落地动作放在 review 或额外验证之后。

这件事很关键，因为它直接决定了第十章的可信度来自哪里：

**不是来自夸大已有能力，而是来自明确说明“今天能安全做到哪一步”。**

#### 10.2.7 第七步：什么叫这轮闭环真的完成

在这个案例里，一轮“成功的自修复”至少要同时满足下面 4 条：

1. 漂移信号被明确定位，而不是只模糊地“好像有问题”。
2. 候选补丁是最小改动，而不是顺手重写周边叙事。
3. 验证重新通过，或者至少能说明为什么新的引用更可信。
4. 整轮过程留下了预算、审计和结果摘要。

如果这 4 条缺任何一条，那么这次尝试更像“自动生成”，还不能叫“受控演化闭环”。

#### 10.2.8 这个案例真正教会了什么

这整段案例最重要的，不是“以后 OpenFang 会自动修教程”，而是下面 3 个更稳的结论：

1. OpenFang 今天已经拥有把一次低风险修复组织成 workflow run 的底座。
2. 它已经拥有把这次尝试放进审计链和预算门的基础设施。
3. 它仍然缺少更强的问题发现器、验证模板和更细的写入边界。

这才是第十章最该给读者的“获得感”：

**不是一句“未来会很强”，而是清楚说明今天已经有哪 4 块砖、还差哪 3 面墙。**

### 10.3 现实边界：OpenFang 现在能主张什么，不能主张什么

写到这里，第十章最危险的诱惑是开始自我鼓舞：既然已经有 workflow、audit、quota、policy，那是不是离“自己改自己”已经只差一步？

答案恰好相反：正因为我们已经回到真实代码，才更应该明确边界。

#### 10.3.1 今天最该避免的误判

最常见的误判有 4 类：

1. **把会生成 patch 误判成会维护系统。**
2. **把测试通过误判成结构改进。**
3. **把能跑单次 demo 误判成能长期自治。**
4. **把模型能力误判成系统能力。**

这 4 类误判背后，其实分别对应 4 个缺口：

1. 缺长期问题发现。
2. 缺强验证面。
3. 缺治理与止损。
4. 缺工程编排与责任边界。

#### 10.3.2 LLM agent 的现实能力边界

公开基准已经反复说明一件事：

**多文件、跨模块、需要长期上下文稳定的工程维护，依然是 LLM agent 的难题。**

第十章因此必须把目标定得足够克制：

1. 先做 low-risk maintenance，而不是 high-risk architecture rewrite。
2. 先做 candidate change，而不是 autonomous merge。
3. 先做 bounded loop，而不是 open-ended recursion。

如果不承认这一点，第十章就会退回“自动改代码”的幻觉。

#### 10.3.3 为什么验证技术比生成技术更稀缺

对于演化系统来说，真正稀缺的从来不是“再多一个会写 patch 的模型”，而是：

1. 这次变化有没有覆盖到真正的失败面。
2. 测到的通过是不是偶然通过。
3. 变化是否引入了新的隐性脆弱点。
4. 不同验证维度之间是否存在冲突。

这也是为什么 mutation testing、property-based testing、golden test、shadow run 这类技术，对第十章来说比“生成更大 patch”更重要。

自我演化系统最终比拼的，不是谁更会动刀，而是谁更会判断哪一刀应该留下。

#### 10.3.4 所以 OpenFang 当前最合理的主张是什么

如果把边界写成一句非常克制、但可信的话，那应该是：

**OpenFang 当前最合理的目标，不是自动重构核心架构，而是安全地承担越来越多低风险、可验证、可审计的工程维护劳动。**

这包括：

1. 文档与教程修复。
2. 配置模板同步。
3. 低风险脚本修正。
4. 测试失败后的候选补丁生成。
5. 高噪音维护劳动的自动总结与建议。

这比“自己重写自己”更慢，但也更像真实工程路线。

### 10.4 双轴校准：OpenClaw 与 ZeroClaw 把这条路校准到哪里

如果只站在 OpenFang 内部看第十章，它很容易继续往“大而全的演化蓝图”滑过去。双轴校准的作用，就是把这种冲动拉回现实，让“系统参与自身演化”这件事重新落到真实工程口径上。

#### 10.4.1 OpenClaw 这条轴提醒的是产品现实压力

OpenClaw 的压力不在于它概念上更先进，而在于它更早地站进了真实渠道、真实消息噪音、真实升级脏活的世界里。

这条轴对第十章最重要的提醒是：

1. 真正值得自动化接手的，不是论文式难题，而是高频脏活。
2. 真正有价值的闭环，不是单次展示能力，而是长期节省维护劳动。
3. 如果系统不能稳稳处理配置漂移、文档漂移、消息碎片和升级副作用，那谈“架构自演化”就会显得空泛。

因此 OpenClaw 轴把第十章的优先级压得更现实：**先修维护劳动，再谈战略级重构。**

#### 10.4.2 ZeroClaw 这条轴提醒的是运行时底线压力

ZeroClaw 的价值也不在于它覆盖更多功能，而在于它不断逼问：

1. 这条回路够不够短。
2. 状态够不够还原。
3. 资源边界够不够硬。
4. 出错时能不能快速失败。

对第十章来说，这条轴会把“自我演化”压成更克制的工程口径：

1. 小步。
2. 可停。
3. 可测。
4. 可回退。

所以 ZeroClaw 轴不是让第十章变保守，而是让它变可信。

#### 10.4.3 放回双轴之后，哪些该补，哪些该借，哪些该坚持

这时第十章终于可以做出比过去更硬的工程判断。

**必须补的：**

1. 稳定的问题发现器，例如教程漂移、配置漂移、接口漂移检查器。
2. 演化专用审计语义，例如 `attempt / verdict / rollback / evidence_insufficient`。
3. 更强的验证模板，把结构、行为、资源、审计这几类门固定下来。
4. 比工具级权限更细的写入边界，尤其是目录级或工作区级变更限制。

**值得借的：**

1. Dependabot / Renovate 这种小步、可 review 的变更模型。
2. 大规模代码变更系统的“先验证后放量”思路。
3. 测试与验证领域里 mutation / property-based 的方法论。

**应该坚持的：**

1. 工作流驱动，而不是开放式自由发挥。
2. 审计链先行，而不是改完再补记录。
3. 预算门先行，而不是“先试试看花多少”。
4. deny-wins 和最小权限，而不是为了方便先放开再说。

把这三组判断压成一句话就是：**OpenFang 第十章真正该推动的，不是更大的自主权，而是更成熟的受控闭环。**

### 10.5 行动项：从哪一代开始，按什么顺序推进

如果把前面的判断最后压回 roadmap，我认为最合理的路径仍然是分代推进，而不是一步跳到“自我重构核心架构”。

#### 10.5.1 Generation 0：文档、教程、配置示例自修复

这一代的目标只有一个：

**先把最容易验证、最容易回滚、最不容易炸主系统的维护劳动接起来。**

最适合进入这一代的对象包括：

1. 教程行号漂移。
2. 文档函数名漂移。
3. 配置示例过期。
4. README / 注释与实现不同步。

这一代最重要的不是“修得多聪明”，而是建立四样东西：

1. 稳定信号源。
2. 最小补丁。
3. 验证门。
4. 审计链。

#### 10.5.2 Generation 1：低风险代码维护工

当 Generation 0 稳定之后，下一代才是：

1. 低风险脚本修正。
2. 重复代码收敛建议。
3. 配置与模板升级。
4. 简单测试失败的候选补丁。

注意关键词仍然是“候选补丁”，不是“自动并入主线”。

#### 10.5.3 Generation 2：受控缺陷修复 agent

到了这一代，系统才开始真正接手更接近工程修复的劳动：

1. 从 test failure / lint failure 生成候选变更。
2. 自动跑验证。
3. 自动给出 evidence summary。
4. 根据预算、权限、验证结果决定是继续、转人工还是终止。

这里最关键的，不是 patch 变大，而是验证链和治理链变硬。

#### 10.5.4 Generation 3：路线图执行者，而不是主分支自治者

只有前面几代都打稳之后，第十章才有资格谈更大的目标：

1. 把第七章的某些行动项拆成可验证小任务。
2. 按 workflow 批次执行。
3. 用验证和审计结果决定下一轮任务。

即使到了这一代，第十章也不应该把目标写成“无人监督的自我递归重写系统”。更成熟的写法应该是：

**让 OpenFang 逐步成长为一个能持续、节制、可回滚地推动自己工程演进的系统。**

### 10.7 本章小结

如果把这一章压成最后几句真正值得带走的话，我会保留下面 5 条：

1. 可演化，不等于会自动改代码，而等于能在边界内持续发现问题、组织变化、验证变化、保留收益。
2. OpenFang 今天已经拥有工作流编排、审计链、预算门、工具级权限门这 4 个真实底座。
3. 它距离真正的演化系统，还差稳定信号源、强验证模板、专用审计语义和更细的写入边界。
4. 真正可信的起点不是高风险架构重写，而是文档修复、配置同步、低风险维护自动化。
5. 第十章真正的方向不是“更大权限”，而是“更成熟的受控闭环”。

到这里，第十章终于不再是“未来想象”。它更准确的定位应当是：

**它给整套教程补上了最后一层工程判断：OpenFang 的终局，不是做一个更会说话的 agent，而是做一个更会组织自己工程代谢的系统。**
