## 附录 A：术语表与核心概念索引

> **本附录命题**：为了解决全书涉及的大量 OpenFang 专有名词和自创概念，本附录提供一份简明的术语索引，方便在跳读或查阅时快速对齐认知。

---

### A.1 系统架构与组件术语

- **OpenFangKernel**：OpenFang 的核心组装结构，管理内存、API、后台线程、渠道桥接等各种关键运行时资源，是所有 Agent 运行的基础底座。
- **MemorySubstrate**：OpenFang 的记忆基底子系统。负责统一长期记忆的所有后台逻辑（包括结构化记忆、语义记忆和会话状态），底层以 SQLite 支撑。
- **SkillRegistry**：统一的工具和技能注册表。不仅管理现有的 Rust/WASM 技能，还负责接收和转化兼容 OpenClaw `.md` / Node 架构的外部工具。
- **LoopGuard**：主循环熔断器。通过 SHA-256 哈希环等检测方式，强制中止 Agent 进入无限自我对话或重复工具调用的死亡螺旋。
- **Compactor**：由于上下文经常被大量报错或长输出撑爆而设立的"压缩器"，它能自动按 Map-Reduce 方式将长会话历史截断或抽象。
- **ChannelBridgeHandle**：各渠道（如 WhatsApp、Discord、Cli）与核心模型通信的系统门面，负责收口消息标准化、分发和跨渠道语义的一致性。
- **EventBus**：OpenFang 的异步事件总线，用来传播系统级状态变化、触发器信号与运行时通知，是第五章治理链的基础骨架之一。
- **MeteringEngine**：面向 token、成本与使用统计的持久账本层。它不是 UI 指标面板，而是配额治理的底层数据来源。
- **AuditLog / Audit Trail**：用于记录高风险动作的审计链路实现。当前核心特征是 hash-chain 可验证性，而不是完整的持久保全体系。
- **WorkflowEngine**：通用工作流编排器，用来把多步 agent 行为收成可执行 run，而不只是简单串联工具调用。
- **TriggerEngine**：触发器解释器。负责把事件、关键词、时间或其他触发条件转成实际工作流启动信号。
- **SemanticStore**：记忆子系统里负责 embedding 写入、候选召回与相似度重排的部分。当前是“SQLite 候选集 + 进程内重排”路线，而不是真正的向量库。
- **SessionStore**：保存会话历史与 canonical session 关系的存储层，用来支撑跨渠道连续性与长期状态恢复。
- **ConsolidationEngine**：记忆衰减与清理逻辑所在的组件。当前更偏 decay，而不是完整的 merge/summarize 夜间归并系统。
- **HandRegistry**：后台 Hands 模板注册表。负责把一组可配置的长期工作模板转换为真正运行的 Agent。
- **ResourceQuota**：资源配额结构体，定义 token、成本、时间或工具调用等维度的上限。它本身是约束声明，不自动等于完整执法闭环。
- **ToolPolicy**：工具策略层，决定哪些执行行为需要允许、拒绝或审批，是第三章执行安全的决策入口。
- **WasmSandbox**：WASM 执行沙箱实现。核心包括 Guest ABI 限制、fuel budget 和超时控制。
- **ShellBleed**：执行前的环境变量泄露扫描层，用来阻断脚本中对高危环境变量的直接暴露。
- **OFP（OpenFang Protocol）**：OpenFang 在节点互联方向上的自定义协议实现，用于补足 HTTP 之外的点对点节点通信能力。

### A.2 设计理念与方法论术语

- **双轴框架（产品现实压力轴 vs 运行时底线压力轴）**：贯穿第四至第九章的核心分析法。前者通常对标 OpenClaw 以代表产品级兼容性与生态广度，后者常对标 ZeroClaw 以要求运行时安全、类型系统和底线不退让。
- **内求 vs 外生**：第十章用来评估框架下一步演化应该向内拆小耦合面（内求），还是向外兼容更大系统交互面（外生）的决策词汇。
- **四平面**：观察平面、执行平面、验证平面与治理平面的切面总结。这也是 OpenFang 在讨论“系统自动修复”和演化时的必须防线。
- **验收卡 / 止损卡**：附录 D 中提到的治理框架。强调系统变更前的质量红线（如行为验收、结构验收等）与退路保证。
- **能力边界（Capability）**：第三章在讨论沙箱时高频使用的术语。与其给工具广泛的网络权限，不如缩小至严格约束的 Host Call 权限隔离单元。
- **结构化分层容错**：第二章反复强调的运行时哲学。与其把全部修复逻辑塞进一个大循环，不如拆成 repair、overflow、compactor、loop guard 等可独立回归的子系统。
- **产品现实压力轴**：借 OpenClaw 观察真实渠道、真实用户和真实升级路径会优先暴露系统哪些薄弱点。
- **运行时底线压力轴**：借 ZeroClaw 观察在更硬的资源约束和更轻的运行形态下，哪些实现显得过重、过松或不够稳。
- **问题发现器**：第十章里的关键概念，指一类负责发现教程漂移、配置漂移、接口漂移等低风险维护问题的自动化组件。
- **受控闭环**：全书后半段反复出现的核心判断。不是指“系统会自己做事”，而是指问题发现、执行、验证、审计、回滚形成了可治理的回路。
- **小步、可停、可测、可回退**：第十章借 ZeroClaw 轴提出的演化原则，用于约束自动化工程系统的推进方式。
- **文档漂移**：实现、注释、教程或配置示例之间逐渐失去同步的现象。本教程本身就把它作为一个真实案例来分析。
- **碎消息防抖**：渠道接入层面对高频小消息、重复消息或分片消息进行重组和节流的能力，主要在第六章作为产品现实压力提出。
- **能力矩阵**：第九章建议引入的控制面表达方式，用来明确 daemon / in-process / MCP 三种后端形态下各功能的可用性差异。

### A.3 生态层术语

- **MCP（Model Context Protocol）**：Anthropic 提出的模型代理上下文与工具通信规范。OpenFang 在第六章、附录 B 提到了针对 MCP 体系的治理与桥接策略。
- **A2A（Agent-to-Agent）**：代理间协同标准协议（在第六章提到）。目前多用于描述在系统内部进行任务转发时的情况。
- **OpenClaw 的 160-pit**：指 OpenClaw 在前一代 Node.js 产品运行中踩过的 160 个业务/安全盲点。常作为 OpenFang 重构成功率的一项量化验证资产。
- **Guest ABI**：WASM 模块与宿主交互时可见的最小接口面。OpenFang 将其收得很窄，只暴露 `memory`、`alloc`、`execute` 等基本入口。
- **Fuel Limit**：WASM 沙箱里的指令预算上限，用来阻断死循环或异常消耗资源的执行体。
- **Epoch-based interruption**：Wasmtime 提供的中断机制，允许宿主在超过资源预算时停止执行中的模块。
- **Wire Protocol**：第六章与第九章提到的节点互联消息格式与握手约定，不等同于 HTTP API。
- **Merkle Hash Chain**：通过把当前记录与上一条记录的哈希值串联起来形成的可验链结构，用于增强审计不可篡改性。
- **Doctor 系统**：启动前健康检查与环境诊断能力的统称。教程里多次把 ZeroClaw 的 doctor 作为对标参照。
- **IntegrationTemplate**：附录 B 中的集成模板定义，用来把第三方能力收束成可安装、可桥接、可治理的结构化集成件。
- **CredentialVault**：附录 B 提到的凭证保险箱组件，用于以受控方式保存外部集成所需凭据。
- **PKCE Flow**：OAuth 场景里的授权流程变体，附录 B 讨论它在本地回调安装流中的实现价值。

### A.4 教程内高频判断句式

- **“不是有没有，而是闭环够不够硬”**：本教程对 OpenFang 的典型评价方式。表示系统通常已有实现，但治理、验证或恢复链还没完全闭合。
- **“主线 + 双轴 + 场景层”**：本教程独有的阅读框架。主线看 OpenFang 自身实现，双轴借 OpenClaw/ZeroClaw 施压，场景层判断差距在哪类真实使用里会出事。
- **“系统骨架”**：用来描述已经存在、但还没完全产品化或工业化闭环的一组底层设施。
- **“长期运行治理链”**：指记忆、预算、审计、调度、控制面等能力最终被收口为一条能支持长期自治运行的治理链路。
