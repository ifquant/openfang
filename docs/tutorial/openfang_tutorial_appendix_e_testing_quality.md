## 附录 E：测试与质量保障——把自动化建立在可验证性之上

> 这份附录把分散在第二、第三、第五和第十章里的验证线索收成一条完整主线。
> 如果说前面的章节解释了 OpenFang 怎样运行，那么这一份附录解释的是：**OpenFang 怎样证明自己没有悄悄变坏，以及为什么未来的自我演化必须站在这条验证链上。**

---

## E.0 为什么这份附录必须存在

整套教程一路讲了很多“能做什么”的能力：

1. 主循环如何容错。
2. 工具如何隔离。
3. 工作流如何编排。
4. 审计和预算如何限权。
5. 系统将来如何参与自己的工程演化。

但如果只讲这些能力，而不把“怎么验证它们没有失控”单独收出来，读者很容易得到一个危险印象：

**好像只要系统足够聪明，就可以逐步接管越来越多的工程劳动。**

真正的工程现实恰好相反。对 OpenFang 这种系统来说，越往后走，越稀缺的不是生成能力，而是验证能力：

1. 你怎么知道容错修复没有引入新的误判。
2. 你怎么知道工具防线没有被变形输入绕过。
3. 你怎么知道工作流自动化没有悄悄扩大爆炸半径。
4. 你怎么知道一轮“自我修复”不是把错误藏起来，而是真的消灭了根因。

因此，这份附录的目标不是另起炉灶，而是把全书里已经存在但分散的质量保障线收成一套更清楚的判断框架。

---

## E.1 四层验证面：OpenFang 到底在验证什么

如果把整套教程里的验证问题压成最重要的骨架，我会把它分成 4 层。

### E.1.1 第一层：runtime 生存验证

这一层主要来自第二章。

它关心的是：

1. 历史消息坏了，系统还能不能继续跑。
2. LLM 回了半截 JSON，系统能不能修回来。
3. 循环卡住时，系统能不能熔断。
4. 上下文爆炸时，系统能不能降级而不是整体崩溃。

这一层的关键词不是“对不对”，而是“能不能活着走到下一步”。

对应的核心模块包括：

1. `session_repair.rs`
2. `loop_guard.rs`
3. `compactor.rs`
4. `context_overflow.rs`
5. `llm_errors.rs`

它们的共同价值，是把 runtime 生存问题拆成了能独立回归的单元，而不是继续堆在 `agent_loop.rs` 里做黑箱补丁。

### E.1.2 第二层：执行安全验证

这一层主要来自第三章。

它关心的是：

1. 工具调用是否越过了不该越过的边界。
2. shell 输入是否包含高风险结构。
3. 文件路径是否真的还在允许边界内。
4. WASM、Docker、subprocess 这些执行路径有没有统一的拒绝与隔离语义。

这一层的关键词不是“运行成功”，而是“即使运行成功，也不能把宿主拖下水”。

对应的核心组件包括：

1. `tool_policy.rs`
2. `tool_runner.rs`
3. `shell_bleed.rs`
4. workspace sandbox
5. `docker_sandbox.rs`
6. `wasm_sandbox.rs`

如果说第二层之前的系统主要担心“活不下来”，这一层开始担心的就是“活下来了，但把边界打穿了”。

### E.1.3 第三层：治理与追责验证

这一层主要来自第五章。

它关心的是：

1. 一次自动化尝试消耗了多少预算。
2. 是谁触发了这次动作。
3. 这次动作有没有留下完整记录。
4. 出问题后，系统能不能说明自己为什么会做出这个决定。

这一层的关键词不是“能不能执行”，而是“值不值得执行，以及事后能不能追责”。

对应的核心组件包括：

1. `MeteringEngine`
2. `ResourceQuota`
3. `AuditLog`
4. `WorkflowEngine`
5. `TriggerEngine`
6. `Supervisor`

这层是 OpenFang 和普通 Agent 框架差异最明显的地方。很多系统会讲“自动化”，但不讲预算、审计、配额和停止条件。OpenFang 至少已经把这些能力做成了真实骨架。

### E.1.4 第四层：演化级验证

这一层主要来自第十章。

它关心的是：

1. 这次变化只是把错误压下去了，还是真的修掉了问题。
2. 这轮自动修复是不是只在单次验证里通过，长期却更脆。
3. 这轮尝试是否应该自动落地、转人工 review，还是直接终止。
4. 这次尝试有没有留下足够多的证据让下一轮少走弯路。

这一层的关键词不是“有没有 patch”，而是“这次 patch 有没有资格留在系统里”。

对未来的 OpenFang 来说，这一层才是真正把“自动化维护”提升成“受控演化”的分水岭。

## E.2 OpenFang 当前已经拥有哪些验证资产

这份附录最重要的任务之一，是避免把 OpenFang 写成“只会谈未来的系统”。从当前源码和教程已有内容看，它已经拥有几类相当扎实的验证资产。

### E.2.1 模块化回归资产

第二章已经说明，`agent_loop` 周围最关键的容错链条已经被拆成独立模块。这个变化的真正价值，不只是代码更清楚，而是：

1. 某一类坏历史只影响 `session_repair` 的回归集。
2. 某一类循环问题只影响 `loop_guard` 的回归集。
3. 某一类压缩失败可以被 `compactor` 单独复现。

这是“可演化”非常重要的一步，因为只有先把失败面拆成可单测、可定位、可复盘的局部，系统才可能安全地让自动化去碰它们。

### E.2.2 分层执行防线

第三章已经证明，OpenFang 的安全不是依赖一个万能沙箱，而是把风险拆到多层：

1. `ToolPolicy` 先做允许 / 拒绝判断。
2. taint tracking 再做高风险前置预警。
3. `shell_bleed` 继续识别环境变量泄露面。
4. workspace sandbox 限定文件边界。
5. WASM / Docker / Remote 再做执行环境层隔离。

这意味着工具安全已经具备了一个很重要的工程属性：

**它可以按层回归，而不是只能整条链端到端碰运气。**

### E.2.3 预算和审计骨架

第五章和第十章合起来，已经把另外两种关键资产讲得很清楚：

1. `MeteringEngine` 可以把成本写入 SQLite 并做时间窗口查询。
2. `ResourceQuota` 已经定义了多维资源上限。
3. `AuditLog` 已经提供了哈希链完整性原型。
4. `WorkflowEngine` 已经能把多步执行收成 run，并保留步骤结果。

这 4 点合在一起的意义是：

**OpenFang 并不是先想好了“将来自动演化很帅”，再回头补治理；它实际上已经先长出了治理骨架。**

### E.2.4 章节级验证出口

现在全书里最重要的验证出口已经明确安放在下面这些位置：

1. 第二章：runtime 生存回归。
2. 第三章：工具与执行边界验证。
3. 第五章：工作流、预算、审计、监督链验证。
4. 第十章：为什么任何自动化维护都必须站在前 3 层之上。

换句话说，这份附录不是凭空新增一个主题，而是把已经存在的质量线显式收出来。

## E.3 当前还缺哪些关键验证能力

把已有资产写清楚之后，更重要的是承认空白。

### E.3.1 缺少统一的问题发现器

第十章已经明确指出，当前系统最大的缺口之一不是“不会生成修复”，而是“没有稳定的问题发现器”。

比如：

1. 教程行号漂移检查器。
2. 文档函数名漂移检查器。
3. 配置示例失效检查器。
4. surface 行为漂移检查器。

没有这一层，后面的自动化修复很容易退化成“模型自己猜哪里不对”。

### E.3.2 缺少更系统的安全测试语料

第三章已经指出，当前防线最大的短板不在“没有闸门”，而在“没有足够系统的语料回归”。

最需要补的至少包括：

1. shell 变形输入语料。
2. `-c` / heredoc / wrapper chain 场景集。
3. symlink / canonicalize 路径拓扑测试。
4. Docker argv 模式与字符串模式对照回归。

同样的思路也应该扩展到渠道层：像第七章 7.4A 讨论的“碎消息防抖”，本质上也需要一套稳定的行为对照样本，否则系统很难证明自己解决的是重复触发，而不是顺手压掉了合法消息。

### E.3.3 缺少统一的演化验收模板落地

附录 D 已经给出了结构、行为、资源、审计四张验收卡，但它们现在仍主要存在于规格层。

真正要让第十章落地，还需要把这些模板变成：

1. CI / workflow 可调用的检查项。
2. 自动化任务的固定输出格式。
3. 转人工 review 时的固定证据包。

### E.3.4 缺少“从验证到放行”的等级制度

当前教程已经明确了很多“要验证”的地方，但还没有完整收出这样一张等级表：

1. 什么任务只需本地检查即可自动落地。
2. 什么任务必须经过人工 review。
3. 什么任务即使测试通过，也只能生成建议补丁。
4. 什么任务当前一律不允许自动触碰。

对自我演化来说，这张表和验证本身同样重要。

## E.4 如何把自动化建立在验证之上

如果把这份附录压成最可执行的部分，我会建议后续所有自动化维护任务都先过下面 4 道门。

### E.4.1 结构门

先问：

1. 这次变化碰的是不是清楚的边界。
2. 新旧路径能不能渐进共存。
3. 这次改动会不会扩大协调节点的爆炸半径。

### E.4.2 行为门

再问：

1. 旧行为是否保持兼容。
2. 失败路径是否明确。
3. 真正的收益是否超过新增复杂度。

### E.4.3 资源门

再问：

1. 预算有没有明确上限。
2. 重试有没有明确上限。
3. 这轮尝试值不值得它消耗掉的资源。

### E.4.4 审计门

最后问：

1. 谁触发了这次动作。
2. 初始信号是什么。
3. 最终结果是什么。
4. 失败时为什么停下。

只有 4 道门都过了，自动化才应该从“候选补丁生成”升级成“可接受的工程动作”。

---

## E.5 最小落地顺序

如果现在要把这条验证线继续往前推，我建议顺序非常克制。

### 第一阶段：先补问题发现器

优先补：

1. 教程 / 文档漂移检查器。
2. 配置示例一致性检查。
3. 关键 surface 行为对照脚本。

### 第二阶段：补验证语料

优先补：

1. shell 变形输入样本库。
2. 路径拓扑测试。
3. 演化任务的最小验收模板。

### 第三阶段：再开放低风险自动化

优先开放：

1. 文档修正建议。
2. 教程引用修正建议。
3. 配置示例同步建议。

这时的关键词仍然是“建议”而不是“自动并入”。

### 第四阶段：把验证结果接回治理层

最后才是：

1. 验收卡接入 workflow 输出。
2. 审计语义补齐 evolution attempt。
3. 预算门和人工 review 门真正并轨。

## E.6 本附录小结

如果把这一附录压成最后几句真正值得带走的话，我会保留下面 5 条：

1. 对 OpenFang 来说，越往后走，越稀缺的不是生成能力，而是验证能力。
2. 第二章、第三章、第五章其实已经分别给出了 runtime、执行安全、治理追责三层验证骨架。
3. 第十章要成立，必须站在这三层骨架之上，而不能跳过它们直接谈自动演化。
4. 当前最该补的不是更大的自治权限，而是问题发现器、验证语料和放行等级制度。
5. 真正成熟的自动化，不是更敢改，而是更会证明自己为什么值得改。

因此，这份附录真正要补上的，是全书最后一条隐藏主线：

**OpenFang 的未来不只是更会行动，而是更会验证自己的行动。**
